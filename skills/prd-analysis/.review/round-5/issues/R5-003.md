# R5-003 — Per-feature observability contracts in nfr.md defeat feature-file self-containment

## Evidence

The new Observability subsection of `common/templates/artifact-template.md` centralises
per-feature contracts inside `architecture/nfr.md` rather than inside the feature file:

1. Lines 264-267 — Metrics Cardinality Limits table is keyed by `Feature`:

   ```markdown
   | Feature | Metric Name | Max Dimensions | Allowed Label Values |
   |---------|-------------|---------------|---------------------|
   | F-001   | <metric>    | ≤ N           | <enumerated or bounded set> |
   ```

2. Lines 284-287 — SLO Definitions table is also keyed by `Feature`:

   ```markdown
   | Feature | SLO Type      | Target                        | Error Budget (30 d) | Burn-Rate Alert |
   |---------|--------------|-------------------------------|---------------------|-----------------|
   | F-001   | Latency P99  | ≤ <N> ms at the 99th percentile | —                  | N/A             |
   | F-001   | Availability | ≥ <99.N>%                      | <M> min / 30 d     | > 5× budget/hr  |
   ```

3. Lines 316-319 — span attributes require per-feature `feature_id: F-NNN` tagging; the same
   applies to the Logging schema (line 339 mandates `feature_id` as a required field).

4. By contrast, the `features/F-NNN-<slug>.md` template (lines 135-193) lists required sections
   `Journey Backreferences`, `Description`, `Acceptance Criteria`, `Inline Data Model`,
   `Inline Conventions`, `Out of Scope` — **no observability section, and no directive to copy
   the feature's SLO / metric budget / span names inline**.

5. The template's own L01 annotation (lines 512-518, negative example) calls out the same
   pattern as a violation:

   > CR-PRD-L01 fires (`feature-files-self-contained`): The feature references
   > `architecture/data-model.md` and `architecture/coding-conventions.md` instead of inlining
   > the relevant content. [...] The file is not independently readable.

## Problem

A coding agent dispatched to implement `F-001 user-login` by reading ONLY
`features/F-001-user-login.md` now has an invisible dependency: the feature's P99 latency
target, metric cardinality budget, span naming obligations, and structured-log required
fields all live in `architecture/nfr.md`. The feature file does not even hint at this —
there is no "see nfr.md for your SLO" line, and the template does not instruct writers to
add one.

This is a textbook CR-PRD-L01 violation introduced by delivery-2:

- CR-PRD-L01 definition (review-criteria.md lines 149-164): "Every feature file [...] MUST
  be independently readable by a coding agent without opening any other file. All
  data-model definitions, relevant coding conventions, and journey context that the feature
  relies on MUST be copied inline into the feature file."
- The observability contract is neither a data-model definition nor a journey context per
  se, but it IS "relevant coding conventions that the feature relies on" — a feature whose
  code emits metrics/spans/logs without knowing its own cardinality budget, span prefix, or
  log-event-type schema will violate the observability contract silently at runtime.

Severity `critical`: this lands on the skill's cornerstone guarantee. Every PRD generated
under delivery-2 will have feature files that look complete but silently depend on
`nfr.md` for implementation-critical constants. Self-containment — the principal promise
of the pyramid — is broken for every observability-aware feature.

This is distinct from R5-002 (CR-PRD-L04, inter-feature ownership ambiguity): that issue
is about two features racing for the same metric name. R5-003 is about one feature's
information being physically split across two files, even with unambiguous ownership.

## Recommended fix

Pick ONE of the following two paths:

1. **Move the per-feature rows into the feature file (preferred).** Replace the single
   Feature-keyed table in `nfr.md` with:
   - A product-level "Observability Conventions" block in `nfr.md` that defines the
     cardinality cap formula, SLO burn-rate math, span naming pattern, and log-schema
     required fields — all as general rules, no `F-NNN` rows.
   - A new required section `## Observability Contract` in the `features/F-NNN-<slug>.md`
     template (artifact-template.md lines 135-193) that holds THIS feature's metric
     declarations, SLO targets, span names, and log event types inline. A coding agent
     implementing F-001 reads only F-001's file and sees its own P99 target and metric
     budget.

2. **Keep nfr.md central but add a mandatory back-reference directive to the feature
   template.** Add to the feature-file Shape section: "If this feature emits metrics,
   spans, or structured logs, copy the relevant observability contract rows from
   `architecture/nfr.md` inline under a new `## Observability Contract` section — do NOT
   write `see nfr.md`." This satisfies L01 by inlining at emit-time, at the cost of
   duplicating content that can drift from nfr.md.

Path 1 is cleaner and matches how the template already treats data-model and coding
conventions: the architecture topic holds conventions; the feature file holds the
feature-specific instance. Path 2 matches how `nfr.md` already treats perf/security/a11y
(centralised product-wide) but shifts the burden to the writer to copy rows manually.
