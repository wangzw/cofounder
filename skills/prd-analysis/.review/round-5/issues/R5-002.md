# R5-002 — Observability subsection feature-ownership ambiguity (CR-PRD-L04 boundary risk)

## Evidence

`common/templates/artifact-template.md` new Observability subsection introduces
per-feature metric declarations (Metrics Cardinality Limits table, lines
264-274) and per-feature SLO definitions (SLO Definitions table, lines 282-294).

Both tables are keyed by a single `Feature` column:

```markdown
| Feature | Metric Name | Max Dimensions | Allowed Label Values |
|---------|-------------|---------------|---------------------|
| F-001   | <metric>    | ≤ N           | <enumerated or bounded set> |
```

```markdown
| Feature | SLO Type      | Target                        | Error Budget (30 d) | Burn-Rate Alert |
|---------|--------------|-------------------------------|---------------------|-----------------|
| F-001   | Latency P99  | ≤ <N> ms at the 99th percentile | —                  | N/A             |
```

## Problem

Observability subsection introduces ambiguity about which feature owns a
metric when features share a schema — plausible boundary violation that
requires clarification.

Concretely: two features that operate on the same data entity (e.g.,
`F-001 user-login` and `F-003 password-reset` both reading/writing the
`User` table) may each independently claim ownership of a metric such as
`auth.user_lookup_latency` in their respective `nfr.md` rows. CR-PRD-L04
`feature-boundaries-clear` requires "no two features in `features/` MAY
overlap in responsibility" — duplicate metric ownership is a form of
responsibility overlap not currently disambiguated by the template.

## Recommended fix

Add a disambiguation rule under the Metrics Cardinality Limits subsection
(before the table) stating one of:

- **Owner-feature rule**: each metric name MUST appear under exactly one
  feature row; if multiple features emit the same metric, the owner is the
  feature whose primary write path emits it, and consumer features reference
  rather than redeclare.
- **Shared-metric rule**: metrics emitted by shared infrastructure belong in
  a dedicated `Shared` row (not a feature row), and the architecture
  section names the owning module rather than a feature.

Either rule eliminates the ambiguity. The same pattern should be applied to
the SLO Definitions table where two features serving the same user-facing
surface could both claim the same latency SLO.
