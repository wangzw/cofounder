## Cross-artifact tension

The artifact template and the writer prompt now mandate an Observability
subsection (prescriptive; writers must emit four specific topics) while
CR-PRD-L05 — the criterion the NFR topic is supposed to satisfy — still only
enforces perf / security / a11y coverage. Concrete downstream consequences:

- A generated PRD's `architecture/nfr.md` can include a well-formed
  Observability subsection that is never checked at review time by any
  automated reviewer (script or LLM). No feedback is produced if the subsection
  is malformed, incomplete, or internally contradictory.
- Conversely, a generated PRD's `architecture/nfr.md` that omits the
  Observability subsection entirely will still pass CR-PRD-L05. The only
  backstop is the writer prompt's own MUST language, which has no downstream
  reviewer enforcement and is effectively self-policed.
- The template itself flags this in both directions (line 27 `advisory`; lines
  253-256 "currently advisory (not enforced by CR-PRD-L05); a future delivery
  may promote it to a hard criterion") — so the gap is acknowledged in-file
  rather than hidden. Delivery-2 is coherent as a prescriptive-only stance.

## Severity rationale

`warning`, not `error`. The delivery-2 plan explicitly designs this
prescriptive-only stance as a narrow-change compromise:

> plan.md line 21: "Narrow-change constraint: CR-PRD-L05 remains scoped to
> perf/security/a11y for delivery-2. Promoting Observability to an enforced CR
> would cascade into every writer self-review, cross-reviewer,
> adversarial-reviewer, and reviser — out of scope for a narrow delivery.
> Observability becomes prescriptive-but-advisory via template + writer prompt
> only; a future delivery can harden it into CR-PRD-L05 v2 or a new
> CR-PRD-L07."

No artifact is broken in delivery-2; the tension is intentional and
scope-bounded. The issue exists to make the follow-up work visible to
the summarizer / judge / future planner rather than to force a revise in
round 5.

## Recommended action (deferred — for delivery-3)

Pick ONE of the following two paths in delivery-3 planning:

1. **Extend CR-PRD-L05 to v2**: update
   `common/review-criteria.md` CR-PRD-L05 predicate from
   `perf + security + a11y` to `perf + security + a11y + observability` and
   bump `version` to `2.0.0`. Cascade the changed predicate into:
   - `generate/writer-subagent.md` line 279 "Applicable CRs" note
   - `review/cross-reviewer-subagent.md` CR enumeration (no predicate change
     but version bump must be acknowledged)
   - `review/adversarial-reviewer-subagent.md` hypothesis list
   - `revise/per-issue-reviser-subagent.md` (no action beyond version
     awareness)
   - `common/templates/artifact-template.md` lines 247-249 — drop "three"
     and the prescriptive/advisory disclaimer (lines 253-256)
   - Writer self-reviews of `nfr.md` must now check observability as a
     PASS/FAIL line under CR-PRD-L05.

2. **Introduce CR-PRD-L07 observability-requirements-present**: add a new
   criterion in `common/review-criteria.md` scoped exclusively to the four
   Observability topics enumerated in the template (cardinality limits, SLO
   templates, tracing-span naming, structured-log schemas). Preserves
   CR-PRD-L05 v1.0.0 semantics; additive rather than breaking. Cascade
   changes are identical to option 1 but target the new CR row instead of
   bumping L05's version.

Option 2 is preferred — additive CR extensions avoid CR-version churn and
keep the reviewer dependency graph stable. Option 1 couples all four
dimensions to a single criterion which makes future dimension-specific
failures harder to diagnose.
