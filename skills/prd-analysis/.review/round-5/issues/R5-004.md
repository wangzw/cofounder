# R5-004 — Observability log/SLO subsections overlap Security and Performance responsibilities inside the same nfr.md

## Evidence

The new Observability subsection prescribes schemas that duplicate responsibilities already
owned by the existing perf and security dimensions of `nfr.md`:

1. **Auth telemetry overlap (Observability ↔ Security).** Lines 345 (extension table):

   ```markdown
   | `auth_event`    | `event_kind` (`login|logout|token_refresh|mfa_challenge`),
                       `outcome` (`success|failure`), `user_id_hash` |
   ```

   But `nfr.md` already owns Security: artifact-template.md line 245 says the file covers
   "security posture (auth, encryption, OWASP coverage)." A writer now has TWO subsections
   inside the same file both dictating what auth events look like — Security's auth posture
   (policy: rate-limits, lockouts, MFA enforcement) and Observability's `auth_event`
   schema (structure: event_kind enum, outcome enum, hash discipline). The enum values
   `login|logout|token_refresh|mfa_challenge` ARE the security vocabulary; the template
   does not state which subsection owns that enum. A writer editing Security's auth enum
   to add `sso_login` and forgetting to update Observability's schema (or vice versa)
   produces inconsistent policy/telemetry — the exact drift CR-PRD-L04 exists to prevent.

2. **Latency measurement overlap (Observability ↔ Performance).** Lines 286, 345:

   ```markdown
   | F-001   | Latency P99  | ≤ <N> ms at the 99th percentile | ...
   ```

   ```markdown
   | `http_request`  | `method`, `path`, `status_code`, `duration_ms` |
   ```

   `nfr.md` already covers "Performance targets (p95 latency, throughput)" (line 245). Now
   Observability declares latency P99 SLOs and defines `duration_ms` as the http_request
   measurement. Performance says p95, Observability says P99 — two subsections of the
   same file prescribe latency targets at different percentiles. A writer asked "what's
   the latency target for F-001?" must look in both subsections and reconcile. CR-PRD-L04
   (`feature-boundaries-clear`) is defined over feature files, but the same principle
   applies to subsection boundaries within one file: "Each [scope] MUST have a distinct,
   non-overlapping set of behaviors."

3. **PII redaction overlap (Observability ↔ Security).** Line 353:

   > No raw PII in log payloads; hash or redact user identifiers before logging.

   PII handling and hashing are classical security-posture concerns. The template already
   earmarks them for Security (line 245: "security posture (auth, encryption, OWASP
   coverage)" — OWASP covers A02 Cryptographic Failures which encompasses PII redaction).
   Observability now re-legislates the same rule. Where is the source of truth?

## Problem

The Observability subsection was added to `nfr.md` alongside perf / security / a11y without
carving out what it owns versus what it borrows. The result is three concrete overlaps
inside ONE file:

- auth event vocabulary (Observability's `event_kind` enum vs Security's auth posture),
- latency target (Observability's P99 SLO vs Performance's p95),
- PII-in-logs (Observability's "hash or redact" rule vs Security's encryption posture).

CR-PRD-L04 fires on `feature-boundaries-clear`. Although the criterion's text scopes to
`features/`, the template is now prescribing writer behaviour for `architecture/nfr.md` in
a way that will cause in-file responsibility overlap on every generated PRD. The cross-
reviewer issue R5-002 flagged inter-feature boundary ambiguity (two features claiming the
same metric). R5-004 is the complementary intra-file boundary ambiguity: within one
`nfr.md`, four subsections collide.

Severity `error` (not critical): unlike R5-003's self-containment breach, this is
recoverable by a small rule addition to the template and does not silently break coding
agents. It DOES, however, guarantee inconsistency noise in every generated PRD's NFR
section and will produce confusing reviewer output when CR-PRD-L05 (perf / security / a11y)
and the advisory Observability rules are checked separately.

## Recommended fix

Add an "Ownership Matrix" block to the new Observability subsection (before the
Metrics/Cardinality table) that states, for each overlap, which subsection owns which
aspect:

```markdown
## Observability — Scope & Ownership

This subsection prescribes HOW feature behaviour is observed in production. It does NOT
re-state perf / security / a11y policy; where policy and telemetry collide, the owning
subsection is:

| Concern                              | Owner       | Observability role                    |
|--------------------------------------|-------------|---------------------------------------|
| Auth event enum (login, logout, ...) | Security    | Consumer — `event_kind` uses Security's enum verbatim; do NOT redefine here |
| PII redaction rules                  | Security    | Consumer — logging subsection references Security's redaction policy |
| Latency target percentile (p95/p99)  | Performance | Observability expresses SLO at the percentile Performance selects; do NOT pick a different percentile |
| Latency measurement source           | Observability | Defines `duration_ms` and span phase timing |
| Metric cardinality budget            | Observability | Sole owner |
| Error-budget burn rate               | Observability | Sole owner |
| Span naming                          | Observability | Sole owner |
| Log-schema required fields           | Observability | Sole owner |
```

With this matrix, a writer editing Security's auth enum knows Observability's schema
inherits it; a writer adjusting Performance's latency target knows Observability's SLO
must match the same percentile; PII rules have one source. No content overlap remains.
