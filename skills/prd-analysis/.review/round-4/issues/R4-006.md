# CR-L11 criteria-internally-consistent — issue_id frontmatter format diverges across cross-reviewer / adversarial-reviewer / per-issue-reviser sub-agent prompts

## Finding

The three review/revise sub-agent prompts each prescribe a different frontmatter key and format for issue files. Since all three roles WRITE to the same `<target>/.review/round-<N>/issues/` directory, downstream consumers (judge, summarizer, aggregate.py) cannot rely on a single schema. This is a criteria-consistency concern because the issue-frontmatter schema IS part of the review-protocol contract governed by CR-L11.

## Evidence

Format variants in the three prompts:

```
review/cross-reviewer-subagent.md line 202:
    issue_id: prd-analysis-round-<N>-<seq>

review/adversarial-reviewer-subagent.md line 305:
    issue_id: <target-slug>-round-<N>-<seq>

revise/per-issue-reviser-subagent.md line 230 (meta-issue schema):
    issue_id: <prd-slug>-round-<N>-<next-seq>
```

And an additional independent variant in actual filesystem usage: round-2 issue files on disk use `id: R<N>-<seq>` (e.g. round-2/issues/R2-001.md frontmatter line 2 reads `id: R2-001`) — a FOURTH format that none of the three sub-agent prompts specifies.

## Why this violates CR-L11

CR-L11 criteria-internally-consistent is the guardrail against review-protocol fragmentation. The issue-file schema is the interface between reviewer → reviser → judge → summarizer; four parallel schemas guarantee that at least one of them breaks consumer parsers. Checker scripts (`check-frontmatter.sh`, `extract-criteria.sh`) and aggregate.py will each pick one of the four interpretations and silently miscount issues that use the other three.

## Suggested remediation

Pick ONE canonical format and update all three sub-agent prompts AND align with the on-disk convention. The on-disk convention `R<N>-<seq>` (used by round-1/round-2 issue files) is the cheapest to converge on:

```yaml
---
id: R<N>-<seq>                     # zero-padded 3 digits
round: <N>
file: <target-relative-path>
criterion_id: <CR-Lxx | CR-Sxx | CR-META-xxx>
severity: critical | error | warning | info
source: cross-reviewer | adversarial-reviewer | per-issue-reviser | self-review-escalation
reviewer_variant: cross | adversarial | meta | null
status: new | persistent | resolved | regressed
---
```

Update:
- cross-reviewer-subagent.md §Output Contract lines 196–211 to use `id: R<N>-<seq>` (drop `prd-analysis-round-` prefix).
- adversarial-reviewer-subagent.md §Issue File Schema lines 294–313 to use the same `id:` key and format.
- per-issue-reviser-subagent.md §Meta-Issue Schema lines 226–243 to use the same `id:` key and format.

This collapses four schemas to one and aligns with the historical on-disk usage round-1/round-2 already applied.
