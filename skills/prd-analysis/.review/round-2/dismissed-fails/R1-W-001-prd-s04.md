# Dismissal — R1-W-001 PRD-S04 Reclassification

## Context

Writer R1-W-001 authored `common/review-criteria.md` and produced a PARTIAL self-review
with one FAIL row citing `CR-PRD-S04-reclassification` under `blocker_scope:
needs-human-decision`. The writer reclassified the original R-005 structural criterion
`PRD-S04 feature-files-self-contained` from `checker_type: script` to `checker_type: llm`
and renamed it `CR-PRD-L01`, promoting the originally numbered R-006 LLM criteria into
`CR-PRD-L02..CR-PRD-L06`. The writer flagged the choice as a naming-convention decision
deserving human confirmation.

## Reasoning

The reclassification is sound domain reasoning that the cross-reviewer endorses:

1. The original R-005 placement labeled the criterion `checker_type: script`, but the
   criterion's verification logic — "feature file MUST contain all relied-upon data-model
   fragments and conventions inline, even without explicit file-path references" — cannot
   be satisfied by mechanical regex. An LLM must read the file contents and reason about
   implicit concept dependencies. Keeping `checker_type: script` would have been an error
   visible on first check (the script would have had no deterministic way to implement
   the check, producing a trivial pass or a false-positive sweep).
2. The id-prefix convention in this skill is `S<NN>` for script-type and `L<NN>` for
   llm-type criteria. Keeping `PRD-S04` with `checker_type: llm` would have forced
   downstream readers to parse the id prefix as a misleading signal. Renaming to
   `CR-PRD-L01` aligns id prefix with checker_type, which is the project's established
   convention observed in `skills/skill-forge/common/review-criteria.md`.
3. The id gap in the PRD-S series (PRD-S01..S03, PRD-S05..S08 with PRD-S04 intentionally
   absent) is explicitly documented in the artifact's "About" section, so no future
   reader will be confused about the missing number.
4. The downstream impact on existing downstream artifacts is bounded: only the
   domain-consultant-subagent.md references the old numbering (see a separate issue
   filed this round, R2-001). That is a documentation drift problem to fix, not a
   reclassification rebuttal.

## Action

Dismiss. The reclassification is a reasonable design decision within the domain writer's
scope that does not require a human escalation. The original `blocker_scope:
needs-human-decision` classification was overcautious — the mechanical evidence (checker
feasibility) and project convention (id prefix = checker_type) make this a defensible
technical call that the cross-reviewer confirms.

A separate issue (R2-001) is filed to address documentation drift in
`generate/domain-consultant-subagent.md`, which still references the pre-reclassification
`PRD-S04` and `PRD-L01..L05` numbering.
