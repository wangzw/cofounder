# IPC role table claims `clarification.yml` at round-N but actual write contract is `round-0/clarification/<ts>.yml`

## Offending text (verbatim)

Line 33 of `generate/domain-consultant-subagent.md` (Snippet D role-mapping table):

> "| `domain_consultant` | 1 write | `.review/round-<N>/clarification.yml` (or scoped clarification path) |"

vs. line 105 (Output Contract) of the same file:

> ```
> <skill-root>/.review/round-0/clarification/<ISO-timestamp>.yml
> ```

Line 108 (Example path):

> "Example path: `docs/raw/prd/2026-04-25-my-app/.review/round-0/clarification/2026-04-25T10-15-00Z.yml`"

## Why this fails

Two mismatches inside the same artifact:

1. **Round number**: the IPC table uses `round-<N>` (variable round number), but the actual contract pins it to `round-0` always. The clarification phase runs before any review round, so `round-0` is correct; the `round-<N>` parameterization in the IPC table is misleading.
2. **Path shape**: the IPC table says `clarification.yml` (single file at the round root). The actual contract says `clarification/<ISO-timestamp>.yml` (a `clarification/` subdirectory containing timestamped files — implying multiple clarifications can co-exist, with the most-recent-timestamp one being authoritative). The IPC table's "or scoped clarification path" parenthetical is too vague to disambiguate.

Downstream consumers (planner, writer) read the IPC role-mapping table to know where to look. Both `generate/planner-subagent.md` lines 31, 60, 83 and `generate/writer-subagent.md` line 134 correctly use the `round-0/clarification/<ISO-timestamp>.yml` form, so they do not actually fail — but the IPC table is the wrong source of truth.

The same misleading mapping appears in `review/cross-reviewer-subagent.md` line 35, `review/adversarial-reviewer-subagent.md` line 24, and `revise/per-issue-reviser-subagent.md` line 34 (all four are focus leaves). I am NOT filing duplicate issues for those — they are mechanical copies of the same Snippet D contents and the fix here cascades. They are noted as part of this issue's class scan.
