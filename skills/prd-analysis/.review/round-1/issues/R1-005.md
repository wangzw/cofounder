# planner-subagent.md R-### field mapping contradicts domain-consultant-subagent.md

## Offending text (verbatim)

Lines 67-74 of `generate/planner-subagent.md` ("Derive the concrete leaf list by reading clarification.yml fields"):

```
| clarification field | Drives |
|---------------------|--------|
| `R-003` (artifact structure) | Confirms pyramid shape; number of mandatory architecture topics |
| `R-004` (interaction modes) | Confirms flags (--review / --revise / --evolve present?) |
| `R-005 / R-006` (CRs) | No effect on plan shape — these inform reviewer/reviser; ignore here |
| Domain-specific data in R-001..R-007 | Journey names, feature names, architecture topics scoped to the product |
```

## Why this fails

`generate/domain-consultant-subagent.md` defines the seven normalized requirements as:

| ID | Definition (from domain-consultant-subagent.md) |
|----|------------------------------------------------|
| R-001 | prd-project-slug |
| R-002 | primary-persona |
| R-003 | journey-list |
| R-004 | feature-seed |
| R-005 | priority-policy |
| R-006 | nfr-applicability |
| R-007 | evolve-baseline-presence |

The planner table claims:
- R-003 = "artifact structure" — wrong; R-003 is the **journey-list**.
- R-004 = "interaction modes" — wrong; R-004 is the **feature-seed**. (`interaction modes` is a Touchpoint-table column value enumeration, not a clarification field at all.)
- R-005 / R-006 = "CRs" — wrong; R-005 is **priority-policy**, R-006 is **nfr-applicability**. Neither is "CRs" (criteria-registry).
- R-005 / R-006 are described as "no effect on plan shape" — but R-006 (nfr-applicability) is the SOLE driver of which conditional architecture topics (`design-tokens.md`, `accessibility.md`, `auth-model.md`, `privacy.md`, etc.) appear in the plan. Marking R-006 "ignore here" directly contradicts the planner's own conditional-topic logic in lines 138-148.

This is a critical cross-artifact bug: the planner reading clarification.yml will not be able to interpret R-### fields correctly. The bug is detectable only by reading planner-subagent.md and domain-consultant-subagent.md side-by-side (cross-artifact view), which is the cross-reviewer's job. This is the canonical class of bug that writer self-review cannot catch (each writer sees only its own file).

## Class-based scan results

Single instance — only the planner has this mapping table. The bug is unique to this file.

## Suggested fix

Replace lines 69-74 with the corrected mapping:

```
| clarification field | Drives |
|---------------------|--------|
| `R-001` (prd-project-slug) | Product slug; dated directory name |
| `R-002` (primary-persona) | Persona context for journey derivation |
| `R-003` (journey-list) | Number and naming of `journeys/J-NNN-*.md` leaves |
| `R-004` (feature-seed) | Seed list of `features/F-NNN-*.md` leaves; MVP/roadmap split |
| `R-005` (priority-policy) | Priority rationale framework — informs feature priority labels |
| `R-006` (nfr-applicability) | Which conditional `architecture/{topic}.md` topics to include (design-tokens, accessibility, auth-model, privacy, etc.) |
| `R-007` (evolve-baseline-presence) | FromScratch vs. NewVersion mode selection |
```
