## Why this fails

The two reviewer variants emit issues into the same directory (`.review/round-<N>/issues/`), and downstream consumers (judge, summarizer, per-issue-reviser, run-checkers.sh) match issues by frontmatter key. Cross-reviewer uses `id:`; adversarial-reviewer uses `issue_id:`. Two different keys for the same conceptual field means:

- The judge's persistence tracking (guide §9.3: "Match on `criterion_id` + `file` combination for persistence tracking" — but the issue ID itself uses two different field names, so any tooling that reads `id` will silently skip adversarial issues).
- The reviser's ACK contract (line 110-111 of `revise/per-issue-reviser-subagent.md`: "Issue-id format: `R<N>-<NNN>`...") implicitly treats them as one ID space, but accessing the ID via different keys is fragile.
- run-checkers.sh and the broader IPC log expect a single canonical key.

The cross-reviewer spec also says: "Issue ID format: `R<N>-<seq>` ... consistent with script-emitted issues from `run-checkers.sh`". The script-tier and cross-reviewer-tier use `id`; the adversarial-tier diverges as `issue_id`.

## Class-based scan results

The mismatch is between exactly two focus leaves. Both reviewer prompts are focus leaves; the bug is detectable only by reading them side-by-side.

## Suggested fix

Pick one canonical key (recommend `id:`) and apply consistently. Edit `review/adversarial-reviewer-subagent.md` line 239:

```
issue_id: R<N>-<NNN>
```

→

```
id: R<N>-<NNN>
```

Optionally, add a `<seq>` vs `<NNN>` placeholder consistency check too (cross-reviewer uses `<seq>`, adversarial uses `<NNN>` — both mean "zero-padded 3-digit sequence"; pick one form and document it in `common/review-criteria.md` or in a shared snippet).
