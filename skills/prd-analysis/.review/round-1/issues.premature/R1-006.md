# Domain-consultant prompt role body empty — no clarification questions, no exit condition

## Evidence

The role-specific body of `generate/domain-consultant-subagent.md` (lines 44-54) contains only DOMAIN_FILL sentinels and TODO markers:

```
# domain-consultant-subagent — Domain Clarification Role for prd-analysis

<!-- DOMAIN_FILL: populated by writer-subagent during round 1 -->

**Role**: domain-Consultant (`C` in trace_id). The ONLY role with `user-interaction: true`.

<!-- TODO: describe the domain clarification questions specific to prd-analysis -->
<!-- TODO: list R-001 through R-007 requirements with domain-specific guidance -->
<!-- TODO: specify the output clarification.yml shape for this skill's domain -->
<!-- TODO: describe dialogue behavior and exit conditions -->
```

CR-L07 (reviewer-prompt-discipline) strictly applies to reviewer prompts, but the spirit of normative-language requirements also extends to writer-facing prompts that enforce hard checks. The domain-consultant is a user-facing role with strict exit conditions — these MUST be expressed with normative language.

## Why this fails CR-L07 (and adjacent L06)

While CR-L07 targets reviewer prompts specifically, the domain-consultant prompt has analogous hard-check needs: the dialogue MUST exit only when R-001..R-007 are all `confirmed`. Without this normative statement, the consultant may exit with unresolved requirements, breaking downstream writers who expect confirmed inputs.

CR-L06 adjacency: the consultant prompt would benefit from a "good dialogue" example (short, focused questions; avoids technical jargon) and a "bad dialogue" example (leading questions, multi-part compound questions). Neither is present.

## Suggested fix

Populate the role body with:
1. R-001..R-007 enumeration, each with a MUST-confirmed clause describing what confirmation looks like (e.g., "R-001 product name/slug MUST be a valid filesystem slug").
2. Dialogue behavior: distinction between sparse idea input and `@notes.md` document input; gap-filling pattern (one focused question per turn).
3. Exit condition: "The consultant MUST NOT write `clarification.yml` until every R-00x field has `status: confirmed`. MUST NOT assume defaults for missing R-00x fields."
4. Output shape: `clarification.yml` schema mirroring `round-0/clarification/2026-04-24T10:52:00Z.yml`.
5. Optional: a positive dialogue snippet and a negative (anti-pattern) dialogue snippet.
