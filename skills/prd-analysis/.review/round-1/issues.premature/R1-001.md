# Writer prompt missing quality bar and DO/DON'T examples

## Evidence

The role-specific body of `generate/writer-subagent.md` (lines 47-58) contains only DOMAIN_FILL sentinels and TODO markers:

```
# writer-subagent — Writer Role for prd-analysis

<!-- DOMAIN_FILL: populated by writer-subagent during round 1 -->

**Role**: Writer (`W` in trace_id). Pure-write, no user interaction.

<!-- TODO: describe what "good output" looks like for this artifact domain -->
<!-- TODO: list applicable review criteria with PASS/FAIL examples -->
<!-- TODO: embed the 4 blocker_scope values and self-review discipline -->
<!-- TODO: include a positive example and a negative example -->
<!-- TODO: specify input contract (which files to read before writing) -->
<!-- TODO: specify output contract (artifact path + self-review path) -->
```

No description of "what good output looks like", no positive example (DO), no negative example (DON'T / FORBIDDEN / BAD). The plan entry for this file (`plan.md` line 38) explicitly requires:

> Include one positive example (well-formed F-NNN feature with inlined data model) and one negative example (feature that cross-refs another feature file — violates CR-S04).

Neither is present.

## Why this fails CR-L06

CR-L06 (writer-prompt-quality-bar) mandates that the writer prompt MUST describe "what good output looks like" with at least 1 positive example and at least 1 negative example. The current stub has zero examples and zero quality-bar language. Without grounding, the writer has no anchor for self-review and will produce inconsistent output.

## Suggested fix

Populate the role body with:
1. A "What good output looks like" section differentiated per artifact leaf type (README index / journey / feature / architecture topic).
2. One positive example: a well-formed `F-NNN-<slug>.md` feature spec that inlines its data-model fragment and journey back-reference.
3. One negative example tagged `BAD` / `FORBIDDEN`: a feature file that cross-references `features/F-002.md` by path for its data model (violates CR-PRD-L01 self-contained principle).
4. Applicable-CRs subsection drawn from `in-generate-review.md` (R-005/R-006 criteria).
5. Input contract: list files the writer MUST read (`clarification.yml`, `plan.md`, assigned leaf path).
6. Output contract: 2 writes (artifact leaf + `.review/round-<N>/self-reviews/<trace_id>.md`).
