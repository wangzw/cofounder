# Writer prompt role body lacks blocker_scope taxonomy and self-review instructions

## Evidence

Although the Snippet D IPC header at the top of `generate/writer-subagent.md` includes the 4-value blocker-scope taxonomy table, the **role-specific body** (lines 47-58) does not. The TODO markers explicitly flag this gap:

```
<!-- TODO: embed the 4 blocker_scope values and self-review discipline -->
```

CR-L09 requires that the writer sub-agent's **self-review instructions** list all 4 `blocker_scope` values: `global-conflict`, `cross-artifact-dep`, `needs-human-decision`, `input-ambiguity`. Currently the file has no self-review instructions block at all — only an ACK-format placeholder referencing `self_review_status` and `fail_count`, with zero guidance on how to populate them.

## Why this fails CR-L09

The IPC snippet's presence at the top is insufficient because the writer is instructed to write a separate self-review archive file (see `in-generate-review.md` PASS/FAIL line format). The role body must tell the writer:
1. Which CRs to self-check against (forward-reference to `in-generate-review.md`).
2. How to emit PASS/FAIL rows with a `blocker_scope` value on every FAIL.
3. When a FAIL should trigger `self_review_status=PARTIAL` vs `FULL_PASS` in the ACK.

Without this body content, writers receive no instruction on self-review discipline beyond the IPC header's definition of the 4 values, and may emit FAIL rows without a `blocker_scope` or silently omit the self-review archive.

## Suggested fix

In the role body, add a "Self-Review Discipline" subsection that:
- Explicitly enumerates the 4 `blocker_scope` values with one-line guidance on when to choose each (copy from IPC header but contextualize).
- Defines the PASS/FAIL line format (mirror `in-generate-review.md`).
- States: "Every FAIL row MUST carry exactly one `blocker_scope` value."
- Explains the ACK semantics: `FULL_PASS` when all rows PASS; `PARTIAL` with `fail_count` equal to the number of FAIL rows otherwise.
- Points to `generate/in-generate-review.md` for the per-file-type applicability table.
