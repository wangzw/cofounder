# R6-002 — RESOLVED: Metric ownership rule added, eliminating inter-feature boundary ambiguity

## Prior issue (R5-002)

R5-002 (critical, CR-PRD-L04, cross-reviewer) flagged that the Metrics
Cardinality Limits and SLO Definitions tables were keyed by a single `Feature`
column with no rule preventing two features (e.g., `F-001 user-login`,
`F-003 password-reset`) from independently claiming ownership of the same
metric (e.g., `auth.user_lookup_latency`). The recommended fix was an
owner-feature rule: each metric name appears under exactly one feature row;
shared infrastructure metrics belong in a dedicated `Shared` row.

## Resolution evidence (round-6 snapshot of `common/templates/artifact-template.md`)

The reviser added an explicit **metric ownership rule** to the
`Scope boundaries for nfr.md` block AND inline inside the Metrics Cardinality
Limits subsection. Both placements match R5-002's recommended-fix text.

1. **Lines 273-275 — scope-boundaries block, metric ownership rule:**

   > - **Metric ownership rule** — metrics are owned by the feature that emits
   >   them; shared infrastructure metrics (e.g., database connection-pool
   >   utilisation, message-queue depth) are owned by the architecture and
   >   documented here at the architecture level.

2. **Lines 294-295 — inside the Metrics Cardinality Limits block, owner
   rule restated in the table-rules section:**

   > - Metrics are owned by the feature that emits them; shared infrastructure
   >   metrics are owned by architecture and appear in this table without a
   >   feature column.

3. **Lines 279-281 — the table's own prose makes the single-owner implicit
   contract explicit:**

   > The cardinality budget for every custom metric is declared here at the
   > architecture level, not per feature file. Feature files stay
   > self-contained (CR-PRD-L01); the observability contract is a cross-feature
   > NFR.

## Status rationale

`resolved`. The R5-002 ambiguity scenario (F-001 and F-003 both claiming
`auth.user_lookup_latency`) is now structurally prevented: exactly one owner
feature per metric, and shared-infrastructure metrics have a distinct
representation (no feature column). CR-PRD-L04 (`feature-boundaries-clear`)
responsibility-overlap concern on the Observability tables is closed.
