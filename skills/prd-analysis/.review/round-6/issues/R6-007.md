# CR-L11 criteria-internally-consistent — adversarial-reviewer issue_id schema diverges from canonical R<N>-<seq> format

## Finding

The cross-reviewer-subagent (as revised in round-5, dispatch R5-R-003) now mandates the unified issue-file frontmatter schema with `id: R<N>-<seq>` (cross-reviewer-subagent.md lines 196–217) and explicitly claims "the sibling sub-agent prompts `review/adversarial-reviewer-subagent.md` and `revise/per-issue-reviser-subagent.md` are aligned to the same schema" (lines 199–202). That claim is false: the adversarial-reviewer prompt still uses a different schema.

## Evidence

`review/adversarial-reviewer-subagent.md` lines 316–335 (current, post-round-5):

```
Issue ID continues the same sequence started by cross-reviewer for this round. Before writing,
list the existing round-<N>/issues/ directory, find the highest existing <seq>, and
increment from there. Format: <target-slug>-round-<N>-<seq> (e.g.
prd-analysis-round-1-014).

Frontmatter is YAML; body is structured markdown.

---
issue_id: <target-slug>-round-<N>-<seq>
round: <N>
file: <target-relative-path>
criterion_id: <canonical-CR-id>
...
```

Compare to the canonical cross-reviewer schema (cross-reviewer-subagent.md line 208):

```
id: R<N>-<seq>                     # zero-padded 3 digits; matches filename
```

Two divergences remain in adversarial-reviewer-subagent.md:

1. Frontmatter key is `issue_id:` not `id:`.
2. ID value shape is `<target-slug>-round-<N>-<seq>` (e.g. `prd-analysis-round-1-014`) not `R<N>-<seq>` (e.g. `R1-014`).

## Why this violates CR-L11

CR-L11 criteria-internally-consistent guards the review-protocol contract. The cross-reviewer, adversarial-reviewer, and per-issue-reviser all write to the same `<target>/.review/round-<N>/issues/` directory. Downstream consumers (judge, summarizer, `aggregate.py`) parse a single schema. With two parallel schemas in play:

- `aggregate.py` or any YAML reader keyed on `id:` will miss adversarial-filed issues (which key on `issue_id:`).
- The filename-to-frontmatter round-trip check breaks: cross-reviewer's `R3-007.md` contains `id: R3-007`; adversarial-reviewer's `prd-analysis-round-3-008.md` (per its own instructions) would contain `issue_id: prd-analysis-round-3-008` — two different filenaming schemes in the same directory.
- Cross-reviewer's own prose (lines 199–202) asserts sibling alignment that is demonstrably not the case — a self-inconsistency within the review-protocol contract.

## Suggested remediation

Update `review/adversarial-reviewer-subagent.md` lines 316–335 to match the canonical cross-reviewer schema verbatim:

```
Issue ID format: R<N>-<seq> (zero-padded 3 digits). Continue the same sequence started by the
cross-reviewer this round — list the existing round-<N>/issues/ directory, find the highest
existing <seq>, and increment from there.

---
id: R<N>-<seq>
round: <N>
file: <target-relative-path>
criterion_id: <canonical-CR-id>
severity: critical | error | warning | info
source: adversarial-reviewer
reviewer_variant: adversarial
status: new
---
```

Apply the same change to `revise/per-issue-reviser-subagent.md` §Meta-Issue Schema (line ~230, currently `issue_id: <prd-slug>-round-<N>-<next-seq>`) to complete the alignment.
