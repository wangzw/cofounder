# R6-003 — RESOLVED: Observability declared cross-feature NFR; feature files explicitly forbidden from holding observability content

## Prior issue (R5-003)

R5-003 (critical, CR-PRD-L01, adversarial) identified that per-feature rows
in the Metrics Cardinality Limits and SLO Definitions tables, plus
per-feature `feature_id` tags in span attributes and structured-log schemas,
created an invisible dependency: a coding agent implementing `F-001` by
reading only `features/F-001-user-login.md` would not see its own P99
latency target, metric cardinality budget, span naming obligations, or
structured-log required fields — those lived in `architecture/nfr.md`
without any back-reference from the feature template. The recommended fix
offered two paths: (1) move per-feature rows into the feature file; (2)
keep nfr.md central but mandate back-reference inlining in feature files.
Path 1 was flagged preferred.

## Resolution evidence (round-6 snapshot of `common/templates/artifact-template.md`)

The reviser chose a cleaner third path that dissolves the tension at the
definition level: observability is declared **not a feature-level concern
at all** — it is a cross-feature NFR owned by architecture. Feature files
are explicitly forbidden from carrying observability content, so the
"F-001 reads only its own file but misses its SLO" scenario cannot arise by
construction.

1. **Lines 200-203 — feature-file Shape, new prohibition paragraph:**

   > - **Feature files MUST NOT include observability constraints** (metric
   >   cardinality limits, SLO definitions, span naming, log schemas).
   >   Observability is a cross-feature NFR and lives exclusively in
   >   `architecture/nfr.md`. Placing observability content in feature files
   >   would force cross-references to other features to resolve cardinality
   >   budgets, violating CR-PRD-L01.

   This is the structural resolution: the template declares observability
   is NOT part of "what a coding agent implementing this feature needs to
   know inline." It is a product-wide NFR, analogous to how accessibility
   and security posture also live in `nfr.md` rather than per-feature.

2. **Lines 279-281 — Metrics Cardinality subsection echoes the same
   reasoning for the positive side of the rule:**

   > The cardinality budget for every custom metric is declared here at the
   > architecture level, not per feature file. Feature files stay
   > self-contained (CR-PRD-L01); the observability contract is a
   > cross-feature NFR.

3. **Lines 496-498 — positive example (F-001 user-login) annotation
   explicitly notes the no-observability-in-feature-file property as a
   CR-pass:**

   > - No observability content in the feature file — observability belongs
   >   in `architecture/nfr.md` ✓

4. **Line 222 — architecture.md topic-table description updated so the
   reader of the index sees `nfr.md` owns observability:**

   > | [nfr.md](architecture/nfr.md) | Performance targets, security posture,
   > accessibility requirements, observability requirements |

## Status rationale

`resolved`. R5-003's core concern — "feature file silently depends on nfr.md
for implementation-critical constants" — is dissolved rather than patched:
observability is now defined as architectural-level, not feature-level, so
a self-contained feature file by design does NOT need observability inline.
The CR-PRD-L01 self-containment guarantee for feature files is restored and
strengthened by an explicit negative rule. Persistence-tracking key
`(criterion_id=CR-PRD-L01, file=common/templates/artifact-template.md,
axis=observability-self-containment)` now closes.
