# R6-004 — RESOLVED: Intra-file subsection boundaries disambiguated by Scope boundaries for nfr.md block

## Prior issue (R5-004)

R5-004 (error, CR-PRD-L04, adversarial) identified three concrete
cross-subsection overlaps inside `nfr.md` once the Observability subsection
was added:

1. **Auth telemetry overlap** — `auth_event` `event_kind` enum
   (`login|logout|token_refresh|mfa_challenge`) duplicates the Security
   section's auth-posture vocabulary.
2. **Latency measurement overlap** — Observability's Latency P99 SLO
   vs Performance's p95 latency target inside the same file, with no rule
   on which percentile wins or whether they must match.
3. **PII redaction overlap** — Observability's "No raw PII in log
   payloads; hash or redact user identifiers" duplicates Security's
   OWASP/PII posture.

Recommended fix: add an Ownership Matrix stating, for each overlap, which
subsection owns which aspect.

## Resolution evidence (round-6 snapshot of `common/templates/artifact-template.md`)

The reviser added a **Scope boundaries for nfr.md** block (lines 262-275)
that disambiguates each of the three overlaps R5-004 flagged, at the exact
prose-level of the recommended Ownership Matrix. Each overlap now has a
single owner and the other subsection is a consumer/reference.

1. **Auth enum / security vocabulary — Security owns, Observability
   references (via the log-schema subsection):**

   Lines 270-272 (Security ownership clause):

   > - **Security section** — covers log redaction policy, audit-trail
   >   requirements, and PII handling in logs. Do NOT duplicate these concerns
   >   in the Observability subsection; reference the Security section
   >   instead.

   Plus lines 348-349 (inside the Structured Log Schemas subsection):

   > All application logs MUST be emitted as structured JSON. Each event type
   > requires a declared schema with mandatory fields to enable consistent
   > indexing and alerting. Log redaction policy and audit-trail requirements
   > are a Security concern — see the Security section of `nfr.md`.

2. **Latency target percentile overlap — Performance owns raw targets;
   Observability expresses SLOs derived from them:**

   Lines 264-269 (Observability subsection scope clause):

   > - **Observability subsection** — covers the instrumentation contract: [...]
   >   SLO targets live here because SLOs are an observability concern (they
   >   define how the system is observed and alerted on). Latency *targets*
   >   (the raw p95/p99 thresholds that drive SLOs) are shared with the
   >   Performance section; cross-reference `performance.md` (or the
   >   Performance subsection of `nfr.md`) for the authoritative latency
   >   budget.

   Plus lines 301-303 (inside the SLO Templates subsection):

   > SLO definitions live here in the Observability subsection; see the
   > Performance subsection of `nfr.md` for the underlying latency targets
   > that these SLOs are derived from.

3. **PII-in-logs overlap — Security owns redaction policy; Observability's
   log rule references it rather than restates it:**

   Line 377-378 (inside the Structured Log Schemas rule list):

   > - No raw PII in log payloads; hash or redact user identifiers before
   >   logging (see Security section for the full redaction policy and
   >   audit-trail requirements).

   The parenthetical "see Security section for the full redaction policy"
   converts the prior duplicate rule into a consumer reference, matching
   R5-004's recommended-fix "Consumer — logging subsection references
   Security's redaction policy" row.
